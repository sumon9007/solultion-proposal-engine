# Rule: Writing Standards

These rules apply to ALL content written for proposals in this workspace. They are non-negotiable.

---

## Language

- Write in **British English** by default (colour, organisation, recognise, programme)
- Switch to **American English** only if the client is a US organisation and the user instructs it
- Do not mix British and American spelling in the same document

## Voice

- Use **active voice** in all executive-facing sections (Executive Summary, Next Steps)
- Passive voice is acceptable in technical sections where the actor is irrelevant
- Use first person plural ("we will", "our team") for company actions
- Use second person ("you", "your organisation") to address the client

## Banned Phrases

Never use these phrases in any client-facing section:

| Banned | Replace with |
|--------|-------------|
| leveraging | using |
| synergies | [specific benefit] |
| cutting-edge | [specific technology name] |
| best-of-breed | [specific product name] |
| world-class | [specific evidence or metric] |
| next-generation | [what specifically is new] |
| holistic | [what is included] |
| robust | resilient / reliable / fault-tolerant |
| seamless | [describe the actual integration] |
| We are pleased to present | [start with the client's problem] |
| Thank you for the opportunity | [start with the client's problem] |
| It is our pleasure | [start with the client's problem] |
| digital transformation journey | [specific outcome] |
| end-to-end | [what specific scope] |
| granular | detailed |
| bespoke | tailored / custom |

## Specificity

- Every benefit claimed must be specific, not generic
- Quantify wherever possible: hours saved, percentage reduction, number of users, days to delivery
- If you cannot quantify, describe the mechanism: "by eliminating manual reconciliation, your finance team..."
- Avoid: "this will significantly improve efficiency"
- Prefer: "this will reduce the time to generate monthly reports from 3 days to 4 hours"

## Hedging

Do not use hedging language in client-facing sections:
- Never: "we hope", "we believe", "it is anticipated", "it is expected"
- Prefer: "we will", "this delivers", "the solution provides"
- Where uncertainty genuinely exists, capture it in Assumptions, not in the main body

## Formatting

- Use `##` for section headings in proposals
- Use `###` for sub-section headings
- Use tables for pricing, deliverables, and scope comparisons
- Use numbered lists for sequenced steps
- Use bullet points for unordered items (but prefer prose when there are fewer than 3 items)
- Bold key terms on first use: **Zero Trust**, **Landing Zone**
- Do not bold for emphasis in running prose — rephrase to make the point land naturally

## Document Length

- Executive Summary: 100–200 words
- Understanding of Requirements: 150–300 words
- Proposed Solution: 200–500 words (more for complex solutions)
- Scope of Work: as detailed as needed — precision matters more than brevity here
- About Us: use the standard template — do not shorten or expand
- Total proposal length: typically 8–20 pages; longer for complex multi-phase projects
