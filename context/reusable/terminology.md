# Proposal Terminology Reference

Standard definitions for terms used across proposals in this workspace. Use these definitions consistently.

---

## Commercial Terms

| Term | Definition |
|------|-----------|
| **Indicative Pricing** | Pricing that is estimated and may change following formal scoping. Always note when pricing is indicative. |
| **Fixed-Price Engagement** | An engagement where the total cost is agreed upfront. Changes are managed via Change Requests. |
| **Time and Materials (T&M)** | An engagement billed based on actual hours or days worked. Provides flexibility but carries budget risk for the client. |
| **Day Rate** | The cost per person-day (8 hours) of professional services. |
| **Change Request (CR)** | A formal document that records and prices changes to the agreed scope. |
| **Statement of Work (SOW)** | A legally binding document defining the scope, deliverables, timeline, and commercial terms of an engagement. |
| **Master Services Agreement (MSA)** | An overarching contract defining the general terms of the commercial relationship. Individual engagements are governed by SOWs under the MSA. |
| **Hypercare** | An intensive support period immediately following go-live, with dedicated resources and heightened monitoring. |
| **Validity Period** | The period during which a proposal's pricing and terms remain guaranteed. Standard: 30 days. |

## Engagement Terms

| Term | Definition |
|------|-----------|
| **Kick-off** | The first formal project meeting at which the project plan, team, roles, and governance are confirmed. |
| **Discovery** | A phase in which the current state is assessed through interviews, workshops, and data gathering. |
| **Current State Assessment** | A documented analysis of the client's existing environment, processes, and capabilities. |
| **Target Operating Model (TOM)** | The desired future state of how the organisation will operate, including technology, processes, and people. |
| **Gap Analysis** | A structured comparison of current state vs. target state, identifying what must change. |
| **Technology Roadmap** | A prioritised, time-phased plan for delivering technology improvements aligned to business goals. |
| **RACI** | Responsible, Accountable, Consulted, Informed — a framework for defining roles in a project. |
| **Milestone** | A significant point in the project that marks the completion of a key phase or deliverable. |
| **Go-Live** | The point at which a new system or service goes into production use. |
| **Rollback Plan** | A documented plan for reverting to the previous state if go-live encounters critical issues. |
| **UAT** | User Acceptance Testing — testing performed by the client to confirm that the solution meets their requirements. |
| **HLD** | High-Level Design — an architectural overview of the solution. |
| **LLD** | Low-Level Design — a detailed technical specification for implementation. |

## Managed Services Terms

| Term | Definition |
|------|-----------|
| **SLA** | Service Level Agreement — the agreed performance standards for a managed service. |
| **OLA** | Operational Level Agreement — internal agreements that underpin SLA delivery. |
| **P1 / Priority 1** | Critical incident — complete or near-complete loss of service. Requires immediate response. |
| **P2 / Priority 2** | High priority — significant degradation of service affecting multiple users or key systems. |
| **P3 / Priority 3** | Medium priority — non-critical issue with a workaround available. |
| **P4 / Priority 4** | Low priority — general request, question, or minor issue. |
| **MTTR** | Mean Time To Resolve — average time to fully resolve an incident. |
| **MTBF** | Mean Time Between Failures — average time between incidents. A higher MTBF indicates greater reliability. |
| **Incident** | An unplanned interruption to or reduction in quality of a service. |
| **Request** | A formal request for a standard service or information (e.g. password reset, new user account). |
| **Change** | A modification to a system or service. Changes are subject to CAB approval. |
| **CAB** | Change Advisory Board — a group that reviews and approves changes to minimise risk. |
| **Service Catalogue** | The agreed set of services included in a managed services contract. |
| **TAM** | Technical Account Manager — a named senior contact responsible for the client relationship. |
| **Monthly Service Report** | A monthly document summarising service performance against SLA targets. |

## Cloud and Technical Terms

| Term | Definition |
|------|-----------|
| **Landing Zone** | A foundational Azure environment with governance, security, and networking pre-configured for workload deployment. |
| **Migration Wave** | A batch of workloads migrated together in a managed sequence. |
| **Lift and Shift** | Moving a workload to the cloud with minimal modification (also called rehost). |
| **Replatform** | Moving a workload to the cloud with minor optimisation (e.g. moving from SQL Server to Azure SQL). |
| **Re-architect** | Redesigning a workload to take full advantage of cloud-native capabilities. |
| **Zero Trust** | A security model that assumes no user or device is trusted by default, requiring continuous verification. |
| **MFA** | Multi-Factor Authentication — requiring two or more verification methods to access a system. |
| **Conditional Access** | Azure AD policies that enforce access controls based on user identity, device state, location, and risk. |
| **Hybrid Identity** | An architecture in which on-premises Active Directory is synchronised with Azure Active Directory. |
| **Tenant** | A dedicated Azure Active Directory instance representing an organisation. |
| **Subscription** | An Azure billing and management boundary containing resources. |
| **IaaS** | Infrastructure as a Service — virtualised compute, storage, and networking in the cloud. |
| **PaaS** | Platform as a Service — managed application platforms (e.g. Azure App Service, Azure SQL). |
| **SaaS** | Software as a Service — fully managed applications (e.g. Microsoft 365). |
